招牌辨識推薦支付系統（BP 專案）

Storefront-Sign Recognition & Best-Cashback Payment Recommendation via LineBot

拍下招牌就推薦高回饋支付 · ViT 分類 90% → 資料強化後 97% · 推論 < 1 秒 · LineBot + AWS 全端部署

走到店門口,拍下招牌傳給 LineBot —— 系統自動分類店家類型,立刻回傳這類店家 回饋最高的 TOP3 支付方式與連結。不需安裝額外 App,小店家也能支援。

設計概念

  • 便捷的使用方式,不需要額外安裝。
  • 小店家也可以支援,提升逛街體驗。
  • 在尚未進入商家前,就先知道高回饋支付方式的資訊。

專案簡介

本專案透過拍攝商家招牌圖片,自動分類店家類型,查詢資料庫中該類別 TOP3 高回饋支付方式,並由 LineBot 回傳推薦卡別與連結。

系統架構

   用戶 LineBot 拍攝商家招牌
            │
            ▼
   Flask App Server(AWS EC2)
            │
            ▼
   ViT(Vision Transformer)分類店家類型
            │
            ▼
   MySQL View 查詢該類別 TOP3 高回饋支付(AWS RDS)
            │
            ▼
   LineBot Flex Message 輪播回傳推薦卡別與連結

系統架構:前後端分離設計,MySQL View 整合複雜的支付規則與跨表查詢。

使用技術

類別技術堆疊
圖像分類ViT(Vision Transformer);選型期比較 ResNet50 / InceptionV3 / MobileNetV2
訓練框架PyTorch
資料策略多時段(早/晚)、多角度、九宮格 + 遮蔽物拍攝擴增
後端部署Flask + MySQL(View 跨表整合),架設於 AWS EC2 / RDS
介面LineBot + FlexMessage(輪播訊息)

成效亮點

  • 推論時間約 1 秒內完成,支援使用者即時體驗。
  • 可擴充到多店家類型。

開發歷程

開發過程不是「一路成功」,而是一連串「遇到問題 → 驗證 → 修正」。以下是幾個關鍵節點:

① 專案整體架構怎麼規劃?

解決方式:為強化使用體驗與系統彈性,架構採前後端分離設計,利於模型佈署與模組擴充;考慮到支付規則的多樣與複雜性,資料庫規劃使用 MySQL 的 View 功能,便於整合與快速查閱跨表資料(架構圖見上方「系統架構」)。

② 什麼樣的模型適合這個主題?

解決方式:聚焦在 Top-1 accuracy 至少 70% 以上、每秒浮點運算次數 10 單位以下的區間,挑選出 ResNet50、InceptionV3、MobileNetV2。

模型選型:準確率 × 運算量的取捨區間。

③ 初期 baseline 模型泛化能力不足

解決方式:增加照片資料,涵蓋早上、晚上等不同場景,有效將 ResNet50、InceptionV3、MobileNetV2 三種模型準確率由 75% 提升至 81～88%。

④ 信用卡與電子支付規則複雜,難以直觀判斷優劣

解決方式:將需要前置任務、限新戶、回饋點數價值小於 1 的選項排除,避免用戶難以理解。

⑤ 連續呈現三個支付方式的文字超過一頁畫面

解決方式:採用 Flex Message 輪播訊息,以左右滑動的方式檢視。

⑥ 準確率無法再向上提升

解決方式:嘗試使用當時剛推出的 ViT —— 辨識效率稍微降低,但準確率穩定提升至 90%。

⑦ 招牌被樹、卡車遮蔽或拍攝角度刁鑽,辨識困難

解決方式:大幅提高照片拍攝數量,以九宮格 + 遮蔽物 + 左側 / 正視 / 右側多角度增加資料豐富度,ViT 準確率提高至 97%。
